How it works

From ticket to draft in one command

Run /triage and the workflow handles everything — fetching, classifying, searching, and drafting. You only review.

1

Fetch open tickets

Pulls open Freshservice tickets from your queue created in the last 24 hours (configurable with --since).

filter_tickets MCP
2

Idempotency check

Scans existing ticket conversations for the cq-bot:v1 marker. Already-processed tickets are skipped immediately.

3

Hard exclusions

Skips tickets from VIP email addresses, those matching security or compliance keywords, or any ticket with attachments.

config/exclusions.yaml
4

Classify the ticket

Runs a strict LLM classifier to determine if the ticket is genuinely asking "where do I go to request X?" — anything ambiguous is skipped.

config/prompts/classifier.md
5

Search local KB

Extracts keywords from the ticket, greps the local kb/ folder, and reads the top matching ClickUp doc. No KB match = skip.

6

Draft the reply

Generates a private note using only information from the KB doc. Includes a DRAFT header, the source doc link, and the idempotency marker.

config/prompts/draft_reply.md
7

Post as private note

Creates a private note on the ticket — invisible to the requester until a human agent reviews it and sends the reply.

create_ticket_note MCP · private: true
8

Print the summary

After all tickets are processed, outputs a table showing what was drafted, what was skipped, and why.

Knowledge Base

Your ClickUp docs, locally searchable

The sync script pulls your ClickUp routing docs down to plain Markdown files so the triage workflow can grep them instantly — no vector DB, no API calls at triage time.

graph LR A([/sync-kb]) --> B[sync_clickup_kb.py] B --> C[ClickUp v3 API] C --> D[List docs by\nfolder ID] D --> E{Local copy\ncurrent?} E -- yes --> F[skip] E -- no --> G[Fetch pages\nand content] G --> H[Write kb/slug.md\nwith frontmatter] style A fill:#02a95c,color:#fff,stroke:none style F fill:#f0f2f5,stroke:#e5e5ea,color:#6e6e73
💡
The sync is incremental — it compares date_updated timestamps and skips docs that haven't changed, so re-running is always cheap.

Each synced doc looks like this

--- clickup_doc_id: abc123xyz title: How to Request a New Laptop url: https://app.clickup.com/v/dc/... date_updated: 2024-03-01T10:00:00+00:00 --- ## Laptop Request Process Submit your request via the IT Hardware portal...

The url field is required — every drafted reply includes this link so the requester can find the full doc.

Safety

Designed to do nothing when in doubt

Every skip condition is logged. Every draft is marked. The system has no way to send a reply — only post a private draft for a human to review.

🚫 Causes a skip

🔁

Already processed

If any prior conversation on the ticket contains <!-- cq-bot:v1 -->, the ticket is skipped. No double-drafts.

⭐

VIP requester

Tickets from email addresses listed in exclusions.yaml are never auto-processed.

🔍

Exclusion keyword match

Any ticket whose subject or body matches a regex in the exclusion list (security, incident, PII, etc.) is skipped.

📎

Has attachments

Tickets with file attachments are skipped — they may contain sensitive data that needs human review.

🤔

Classifier returns false

If the ticket isn't clearly asking "where do I go for X?", the classifier returns false and the ticket is skipped. Ambiguity = skip.

📚

No KB match

If no local KB doc clearly answers the question, the ticket is skipped rather than inventing an answer.

✅ Required before posting

🎯

Classifier explicitly true

The classifier must return "is_routing_question": true. A parse error or any non-true result blocks posting.

🔗

KB doc with URL

A KB doc must be found and must contain a url: field in its frontmatter. No URL = no draft.

🔒

Private note only

Notes are always posted with private: true. There is no code path that posts a public reply.

📋

DRAFT header

Every note starts with **DRAFT — review before sending** so it's unmissable in the Freshservice UI.

🏷️

Idempotency marker

Every note ends with <!-- cq-bot:v1 --> so the system knows this ticket was handled and won't re-draft it.

🚫

No invented content

The draft prompt is instructed to return a null sentinel rather than invent any team name, URL, or routing target not in the KB.

Setup

Up and running in 4 steps

No servers, no databases, no subscriptions. Everything runs locally inside Claude Code.

1

Install Python dependencies

Only three libraries are needed — requests, pyyaml, and python-dotenv.

pip install -r requirements.txt
2

Configure your environment

Copy the example env file and fill in your ClickUp credentials.

cp .env.example .env
# ClickUp personal token (Settings → Apps → API Token) CLICKUP_TOKEN=pk_xxxxxxxxxxxxxxxxxxxx # Your numeric workspace ID CLICKUP_WORKSPACE_ID=12345678 # Comma-separated folder IDs holding your routing docs CLICKUP_DOC_FOLDER_IDS=folder_id_1,folder_id_2
3

Connect the Freshservice MCP

The /triage command calls Freshservice through the mcp__freshservice-mcp MCP server. This must be configured and authenticated in Claude Code before running the command. If it's missing, you'll see MCP tool errors when triaging.

💡
If you're already using Freshservice in Claude Code for other workflows, this MCP is likely already connected.
4

Populate the knowledge base

Run the KB sync before your first triage. This pulls your ClickUp docs into the local kb/ folder — there's nothing to search until this is done.

# In Claude Code: /sync-kb
Usage

Two slash commands, that's it

🔄

/sync-kb

Pulls your ClickUp routing docs into the local kb/ folder. Run this whenever your ClickUp docs are updated.

/sync-kb
🎯

/triage

Processes your open ticket queue and drafts replies. Supports optional arguments to control scope.

/triage # last 24h, up to 10 tickets /triage --limit 5 # cap at 5 /triage --since 48h # look back 48 hours /triage --dry-run # classify only, no writes

Customizing

Three files control everything

🚫

config/exclusions.yaml

Add VIP email addresses to vip_emails or new regex patterns to exclusion_keywords to widen what gets skipped. When in doubt, add to this list.

🎯

config/prompts/classifier.md

Controls what counts as a routing question. Add examples to the in-scope or out-of-scope lists to tune the classifier. Narrower = fewer drafts, more precision.

📂

CLICKUP_DOC_FOLDER_IDS

Add more ClickUp folder IDs to your .env and re-run /sync-kb to expand the KB. More docs = better coverage.

v1 scope

What's deliberately not here

These were left out on purpose — not because they're hard, but because v1 should earn trust before doing more.

📤

Auto-send

Stays draft-only until there's enough history to trust the output rate. Target: ≥70% human send-through before considering auto-send.

🔮

Vector / semantic search

Plain grep works well for short routing queries. Revisit only if recall becomes a real problem in practice.

🔔

Webhooks / scheduled polling

Pure on-demand only. No background process, no always-on daemon. You run it when you want it.

🛠️

How-to & break-fix tickets

The classifier is intentionally narrow. Easy to broaden later by editing classifier.md — but v1 stays focused on routing only.
